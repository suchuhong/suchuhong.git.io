1、TCP 头部

2、为什么需要 TCP 协议？TCP 工作在哪一层？

IP 层是「不可靠」的，保障网络数据包的可靠性，那么就需要由上层（传输层）的 TCP 协议来负责， TCP 协议工作在传输层的可靠数据传输的服务

3、TCP 是面向连接的、可靠的、基于字节流的传输层通信协议。

4、UDP 和 TCP 有什么区别呢？分别的应用场景是？

TCP 和 UDP 区别：

1. 连接

  • TCP 传输数据前先要建立连接。
  • UDP 是不需要连接。

2. 服务对象

  • TCP 是一对一的两点服务。
  • UDP 支持一对一、一对多、多对多的交互通信

3. 可靠性

  • TCP 是可靠交付数据的。
  • UDP 是尽最大努力交付，不保证可靠交付数据。

4. 拥塞控制、流量控制

  • TCP 有拥塞控制和流量控制机制。
  • UDP 网络非常拥堵了，也不会影响 UDP 的发送速率。

TCP 和 UDP 应用场景：

由于 TCP 是面向连接，能保证数据的可靠性交付：

  • FTP 文件传输
  • HTTP / HTTPS

由于 UDP 面向无连接，它可以随时发送数据，既简单又高效，因此经常用于：

  • 包总量较少的通信，如 DNS 、SNMP 等
  • 视频、音频等多媒体通信
  • 广播通信

5、TCP 建立连接的状态

建立连接是通过三次握手而进行的

  • 客户端初始化序号（client_isn），序号置于 TCP 首部的「序号」字段中， SYN 标志位置为 1。
  • 发送给服务端，发起连接，该报文不包含应用层数据，之后客户端处于 SYN-SENT 状态。

  • 服务端收到 SYN 报文，
  • 随机初始化自己的序号（server_isn），填入 TCP 首部的「序号」字段中，
  • 把 TCP 首部的「确认应答号」字段填入 client_isn + 1,
  • 把 SYN 和 ACK 标志位置为 1。
  • 发给客户端，该报文也不包含应用层数据，之后服务端处于 SYN-RCVD 状态。

客户端收到报文后，回应最后一个应答报文， TCP 首部 ACK 标志位置为 1 ，「确认应答号」字段填入 server_isn + 1 ，把报文发送给服务端，这次报文可以携带客户到服务器的数据，之后客户端处于 ESTABLISHED 状态。

第三次握手是可以携带数据的，前两次握手是不可以携带数据的

6、什么是 SYN 攻击？

TCP 连接建立是需要三次握手，假设攻击者短时间伪造不同 IP 地址的 SYN 报文，服务端每接收到一个 SYN 报文，就进入SYN_RCVD 状态，但服务端发送出去的 ACK + SYN 报文，无法得到未知 IP 主机的 ACK 应答，久而久之就会占满服务端的 SYN 接收队列（未连接队列），使得服务器不能为正常用户服务。

7、TCP 断开连接

8、未完成接收队列与已完成接收队列

Linux内核中会维护两个队列：

  • 未完成连接队列（SYN 队列）：接收到一个 SYN 建立连接请求，处于 SYN_RCVD 状态；
  • 已完成连接队列（Accpet 队列）：已完成 TCP 三次握手过程，处于 ESTABLISHED 状态；
